## はじめに

2026年6月、X（旧Twitter）である投稿が話題を呼びました。OpenAI の Peter Steinberger と Anthropic/Claude Code の Boris Cherny が、ほぼ同時期に同じメッセージを発信したのです：

> **「プロンプトを打つ時代は終わった。ループを設計する時代だ」**

AIエージェント（Claude Code、Cursor、Codex など）を日常的に使っている開発者なら、一度は感じたことがあるでしょう——「毎回同じようなプロンプトを打っているな」「もっと自律的に動いてほしいな」と。

この感覚を体系化したのが **Loop Engineering（ループエンジニアリング）** です。本記事では、この新しい開発パラダイムの全体像と、実際のプロジェクトへの導入プロセスを解説します。

## Loop Engineering とは何か

Loop Engineering とは、最小限の人間介入で AI エージェントが自律的にサイクルを反復し、ゴールに到達するまで動き続ける仕組みを設計する実践です。

従来の「プロンプトを書いて実行 → 結果を見て次のプロンプトを考える」という **逐次指示型** から、**ゴール駆動ループ型** へのパラダイムシフトと言えます。

### 6段階ループの解説

Loop Engineering のコアは、以下の6段階で構成されるループです：

```text
LOOP {
  DISCOVER（発見）    → 現状把握、前回の状態復元
  PLAN（計画）       → 次に何をすべきか計画
  EXECUTE（実行）    → 実際の実装や記事生成
  VERIFY（検証）     → テスト、品質チェック、事実確認
  EVALUATE（評価）   → ゴール達成判定
  ITERATE（反復）    → 未達成なら DISCOVER から再開
}
→ DONE（ゴール達成）
```

各フェーズの役割は次の通りです：

| フェーズ | 役割 | 具体アクション |
|---------|------|--------------|
| **DISCOVER** | コンテキスト復元 | メモリファイルの読み込み、コードベース調査 |
| **PLAN** | 行動計画策定 | ユーザーストーリー作成、技術ブリーフ作成 |
| **EXECUTE** | 実装実行 | バックエンド/フロントエンド実装、記事執筆 |
| **VERIFY** | 品質検証 | テスト実行、リンター、事実確認、プライバシーチェック |
| **EVALUATE** | ゴール達成判定 | 条件チェック、次アクション決定 |
| **ITERATE** | 状態更新と再ループ | メモリの更新、ループカウンタ増加、DISCOVER へ戻る |

重要なのは、VERIFY で発見された問題は「報告で終わり」ではなく、**自動的に ITERATE に移行して修正される** 点です。

## 6つの構成要素

Loop Engineering のアーキテクチャは、以下の6つの要素から成り立ちます：

### 1. Automations（自動化）
ループ内の定型処理を自動実行する仕組みです。テスト実行、リンター、ビルド検証などが該当します。人間がトリガーするのではなく、VERIFY フェーズで自動的に発動します。

### 2. Worktrees（並列作業領域）
複数のエージェントが同時に作業できる分離された環境です。バックエンドとフロントエンドの並行開発や、複数の調査タスクの同時実行を可能にします。

### 3. Skills（スキル定義）
エージェントに特定のタスクを実行させるための手順書です。今回の事例で言えば `loop-engineering` スキルがこれに該当し、ループの回し方や終了条件を定義します。

### 4. Plugins（プラグイン）
エージェントの能力を拡張する外部ツール連携です。Playwright によるブラウザ操作や、GitHub CLI によるリポジトリ操作などが含まれます。

### 5. Subagents（サブエージェント）
特化した役割を持つ子エージェント群です。コード調査役（code-researcher）、実装役（backend-builder）、検証役（test-verifier）など、それぞれが専門分野を持ちます。

### 6. Memory（記憶）
ループ間の状態を永続化する仕組みです。何を試したか、何が失敗したか、次に何を試すべきかを記録し、セッションを超えた継続性を実現します。

## 2つのスケールと2つのタイプ

Loop Engineering は、スケールとタイプによって4象限に分類できます。

### 2つのスケール

| スケール | 説明 | 適したタスク |
|---------|------|------------|
| **Single-Agent Loop** | 1つのエージェントが自己改善ループを回す | 小規模リファクタリング、単一ファイル修正 |
| **Fleet Loop** | オーケストレーターが複数エージェントを統率する | 大規模開発、複数機能の並行実装 |

今回の argonpt2loop では **Fleet Loop** を採用し、オーケストレーターが15のサブエージェント（Build 7 + Blog 7 + Infra 1）を統率します。

### 2つのタイプ

| タイプ | 説明 | 使用タイミング |
|--------|------|--------------|
| **Closed Loop** | 境界あり。品質ゲートと終了条件が明確 | 通常の機能開発作業 |
| **Open Loop** | 探索的。エージェントの自由な探索を許容 | 調査・プロトタイピング |

Closed Loop は「全テストがパスし、code-validator がクリーンであること」のような明確な終了条件を持ちます。一方 Open Loop は「この技術の調査結果をまとめてほしい」のような曖昧なゴールにも対応します。

## ループの最大の壁──トークンコストとDeepSeek

Loop Engineering の最大の課題は **トークンコスト** です。ループを回せば回すほどAPIコールが発生し、従量課金が積み上がります。特に VERIFY → ITERATE のサイクルが複数回続くと、反復回数に比例してコストが積み上がります。

この問題に対する現時点の現実解が **DeepSeek** です。DeepSeek は OpenAI / Anthropic と比較して圧倒的に低価格であり、ループの試行錯誤を経済的に許容します。今回の argonpt2loop でも `opencode-go/deepseek-v4-flash` を採用し、ループの反復コストを実用レベルに抑えています。

とはいえ、モデル選びはコストだけではありません。タスクの複雑さに応じてループ内でモデルを切り替える戦略も検討に値します（例：PLAN は高性能モデル、EXECUTE は低コストモデル）。

## 実践例：argonpt2（線形）→ argonpt2loop（ループ）への移行

ここからは、実際に筆者が運用しているプロジェクトでの移行判断を紹介します。

### 移行前（argonpt2）：線形パイプライン

argonpt2 は以下のような **線形パイプライン** で構成されていました：

```text
code-researcher → story-writer → spec-writer
  → backend-builder → frontend-builder
  → test-verifier → code-validator
      ↕                    ↕
    差し戻し             差し戻し

人間承認ポイント: code-researcher→story-writer 間,
                  story-writer→spec-writer 間,
                  spec-writer→backend-builder 間,
                  code-validator 完了後
```

特徴：
- 7つのエージェントが直列に実行される
- 各エージェントの出力が「次のステップ」に引き継がれる
- 各フェーズ間で人間承認が必要
- 問題発生時は手動で差し戻しが必要
- 状態はセッションごとにリセットされる

この構造でも実用的には問題ありませんでした。特に **Maker/Checker 分離** や **品質ゲート** といった良い要素は既に備わっていました。

### 移行判断のきっかけ

以下の3つの課題が、ループ駆動への移行を後押ししました：

1. **手動差し戻しのオーバーヘッド**: テスト失敗時に人間が原因を特定し、適切なエージェントに再指示する必要があった
2. **セッション間の継続性の欠如**: 「前回どこまでやったか」を毎回人間が説明する必要があった
3. **並列実行の非対応**: バックエンドとフロントエンドの同時開発ができず、直列実行による待ち時間が発生した

### 移行後（argonpt2loop）：条件駆動ループ

```text
GOAL: 「全テストがパスし、code-validator がクリーンであること」

LOOP {
  DISCOVER → memory から状態復元 / code-researcher で現状調査
  PLAN     → story-writer + spec-writer → ⏸ 人間承認（初回のみ）
  EXECUTE  → backend-builder / frontend-builder（並列可能）
  VERIFY   → test-verifier → code-validator → fact-checker / privacy-reviewer
  EVALUATE → ゴール達成？ → Yes → DONE
                          → No  → memory 更新 → ITERATE
}
```

**エージェントテンプレートはそのまま流用** し、**オーケストレーション層だけ** を「線形パイプライン」から「条件駆動ループ」に変更しました。これにより、エージェントの定義をゼロから書く必要はなく、既存の資産を活かしたまま移行できました。

## 設計上のポイント

実際にループを設計する際に考慮すべき3つのポイントを紹介します。

### 1. 状態管理

ループ間の状態をどこに保存するかは、アーキテクチャの要です。argonpt2loop では `.opencode/memory/current-session.md` に状態を永続化しています：

```text
# current-session.md（ループ状態の保存イメージ）

## 現在のループ
- Loop: 2/5
- フェーズ: EXECUTE（backend-builder 実行中）
- ゴール: メモ一覧に日付ソート機能を追加する

## 前回の結果（Loop 1）
- VERIFY 結果: テスト2件失敗（日付パースのエッジケース）
- 学び: 空文字列の日付を考慮していなかった
- 次に試すこと: 空文字列ガードを追加して再実行
```

このファイルを DISCOVER フェーズで読み込むことで、セッションを超えた継続性を実現します。

### 2. 終了条件設計

ループが「いつ止まるか」を定義する終了条件は、最も重要な設計要素です。

**良い終了条件の例：**
- 「全ユニットテストがパスし、lint エラーがゼロであること」
- 「fact-checker と privacy-reviewer の両方を通過していること」
- 「hugo build がエラーなく完了し、サーバーが正常起動すること」

**悪い終了条件の例：**
- 「いい感じになったら」
- 「バグが全部取れたら」（バグの定義が不明確）

終了条件は **客観的に測定可能** でなければなりません。主観が入る条件はループが無限に回る原因になります。

### 3. 人間チェックポイント

完全自律は理想的ですが、現実的には **人間の承認が必要なポイント** を設計に組み込む必要があります。argonpt2loop では3箇所に設定しました：

| チェックポイント | タイミング | 理由 |
|----------------|-----------|------|
| 初回 PLAN 承認 | 最初の計画策定後 | 方向性の誤りを早期に発見 |
| PR 作成前 | EXECUTE 完了後 | 意図しない変更の混入防止 |
| 公開 push 前 | 全検証通過後 | 最終的な責任は人間にある |

これらは「人間が全てを決める」のではなく、「重要な判断だけを人間に委ねる」というバランスを取っています。

## 6つの構成要素を argonpt2loop にマッピングする

argonpt2loop の実際の構成と、6つの要素の対応関係を示します：

| 要素 | argonpt2loop での実装 |
|------|---------------------|
| **Automations** | VERIFY フェーズでの自動テスト実行、ビルド検証 |
| **Worktrees** | 未実装（今後の課題：backend/frontend の並列実行） |
| **Skills** | `argonpt2loop/.opencode/skills/loop-engineering/SKILL.md` に定義 |
| **Plugins** | Playwright（ブラウザ操作）、GitHub CLI（リポジトリ操作） |
| **Subagents** | 15のエージェント（Build 7 + Blog 7 + Infra 1） |
| **Memory** | `.opencode/memory/current-session.md` + アーカイブ |

Worktrees はまだ未実装ですが、これは「最初から完璧を目指さない」というLoop Engineeringの現実的なアプローチを反映しています。

## まとめ

Loop Engineering は、AIエージェントの使い方を「プロンプトを打つ」から「ループを設計する」へと根本的に変えるパラダイムです。本記事で紹介した内容をまとめます：

1. **Loop Engineering のコア**: DISCOVER → PLAN → EXECUTE → VERIFY → EVALUATE の5段階ループ
2. **6つの構成要素**: Automations, Worktrees, Skills, Plugins, Subagents, Memory
3. **2軸の分類**: Single-Agent / Fleet Loop × Open / Closed Loop
4. **現実的な課題**: トークンコスト → DeepSeek で緩和
5. **移行戦略**: エージェント定義は流用し、オーケストレーション層だけを置き換える

Steinberger と Cherny の「ループを設計する時代」という言葉は、単なるトレンドではなく、AIエージェントの運用が成熟期に入ったことを示しています。最初の一歩として、**「今日の手動差し戻しを、明日は自動反復にする」** ことから始めてみてはいかがでしょうか。
