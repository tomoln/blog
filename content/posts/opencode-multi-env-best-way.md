## 背景

私は現在、Windows 11でOpenCode v1.17.7をメインの開発環境として使用しています。CPUはIntel i5-1335U、RAMは16GB、空きディスク容量は208GBと、個人開発用としてはまずまずのスペックです。

しかし、複数のプロジェクトを並行して進める中で「もう1つ別のOpenCode環境が欲しい」と感じる場面が増えてきました。同じマシンに複数のOpenCodeをインストールしようとすると、以下のような競合が発生する可能性があります。

- 同じプロジェクトで複数インスタンスを起動した際の `.opencode/` ディレクトリ内の設定・エージェント定義の競合
- セッションデータ（`.opencode/session/` のMarkdownファイル）の混在
- npmグローバルモジュールのバージョン競合

これらを避けるには、**完全に分離された独立した環境** を別途用意するのが最も確実です。本記事では、その方法を4つの選択肢から比較し、最適解を導き出します。

## 比較する4つの選択肢

OpenCodeの第2環境を構築する方法として、以下の4つを比較しました。

| 選択肢 | コスト | 分離度 | 速度 | セットアップ難易度 | ローカルLLM対応 |
|--------|--------|--------|------|-------------------|----------------|
| **WSL2 + Ubuntu** | 無料 | ★★★★★ | ★★★★★ | 簡単（10〜15分） | ✅ 可能 |
| **GitHub Codespaces** | 月120h無料/超過$0.18/h | ★★★★★ | ★★★ | 非常に簡単 | ❌ 不可 |
| **Docker Container** | 無料 | ★★★★ | ★★★★ | やや面倒 | △ 要設定 |
| **別PC / VPS** | $5〜10/月 | ★★★★★ | ★★ | 難しい | △ 要設定 |

それぞれ詳しく見ていきましょう。

---

### 1. WSL2 + Ubuntu（最も推奨）

**結論から言うと、WSL2が最適解です。**

WSL2はWindows上で完全なLinuxカーネルを動作させる仕組みです。OpenCodeの第2環境として以下の点で優れています。

**メリット**
- **コストゼロ**: Windows 11には標準搭載されており、追加費用は一切かかりません
- **完全な分離**: Linuxカーネルが独立した空間として動作するため、Windows側の設定ファイルと競合しません
- **圧倒的な速度**: ほぼネイティブのパフォーマンスが出ます。私のi5-1335U環境でもストレスは皆無です
- **VS Code Remote-WSL対応**: VS CodeからシームレスにWSL内のファイルを編集できます
- **Ollama連携可能**: ローカルLLMを使っている場合でも、`localhost:11434` 経由でWSLからホストOSのOllamaにアクセスできます
- **メモリ制限可能**: `.wslconfig` ファイルでWSLが使うメモリを制限できるため、ホストOSに影響を与えません

**デメリット**
- Linuxコマンドに少し慣れが必要
- WSL2の初期化に10〜15分程度かかる

#### セットアップ手順（WSL2 + OpenCode）

実際の手順は以下の通りです。たった **コマンド3行＋待ち時間** で完了します。

```powershell
# 1. Ubuntuのインストール（管理者権限不要）
wsl --install -d Ubuntu
```

```bash
# 2. Ubuntuの中に入る
wsl -d Ubuntu
```

```bash
# 3. Node.jsのインストール（Ubuntu内）
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt-get install -y nodejs git
```

```bash
# 4. OpenCodeのインストール
npm install -g opencode
```

```bash
# 5. 起動
mkdir ~/workspace && cd ~/workspace && opencode
```

**所要時間**: ダウンロード速度にもよりますが、10〜15分で完了します。

**メモリ制限の設定**（必要に応じて）

Windows側で `%USERPROFILE%\.wslconfig` を作成し、以下のように記述します。

```ini
[wsl2]
memory=8GB
processors=4
```

これでWSL2がホストのメモリを使い切る心配がなくなります。

### 2. GitHub Codespaces

GitHub Codespacesは、クラウド上に開発環境を即座に構築できるサービスです。

**メリット**
- **ブラウザだけで完結**: ローカルに何もインストールする必要がありません
- **どのデバイスからでも**: iPadや Chromebook からでもアクセス可能
- **高い分離性**: 物理的に別のコンテナで動作するため競合は一切ありません
- **プリセット構成**: `.devcontainer/devcontainer.json` で環境をコード管理できる

**デメリット**
- **月120時間の無料枠**: 2コア環境で月120時間まで無料。超過時は $0.18/時間
- **ストレージ制限**: 15GBまで無料、超過時 $0.07/GB/月
- **ネットワーク必須**: オフラインでは使えません
- **ローカルLLMが使えない**: Ollamaなどローカルで動かしているLLMにはアクセスできません
- **速度が回線次第**: 体感速度はネットワーク品質に依存します

**利用シーン**
- 外出先からのサブ環境として最適
- WSL2が使えないデバイス（iPadなど）からの開発に

---

### 3. Docker Container

Docker Desktopを使ってコンテナ内にOpenCode環境を構築する方法です。

```dockerfile
FROM node:22-bookworm
RUN npm install -g opencode
WORKDIR /workspace
CMD ["opencode"]
```

**評価: あまりおすすめしません。**

理由は以下の通りです。

- Docker Desktopの起動が必須であり、起動に時間がかかる
- TUI（ターミナルUI）の操作がやや面倒
- **結局WSL2上で動いている**（Docker DesktopはWSL2をバックエンドに使う）
- ならば最初からWSL2に直接入れた方がシンプルで、トラブルシューティングも容易

どうしてもコンテナで管理したい事情がない限り、WSL2直インストールをおすすめします。

---

### 4. 別PC / VPS

物理的に別のマシン、もしくはVPS（Virtual Private Server）を借りる方法です。

**メリット**
- 最も確実な分離（物理的に別マシン）
- チーム開発時の共有環境として使える

**デメリット**
- **コストがかかる**: VPSで月$5〜10（さくらのVPSなら約500円〜）
- **初期設定が面倒**: SSH鍵の設定、セキュリティ対策などが必要
- **ネットワーク遅延**: どうしてもレイテンシが発生する
- **オーバースペック**: 今回の目的には過剰なリソース

**評価**: 現状のローカルリソースで十分に事足りるため、今すぐ検討する必要はありません。どうしてもWSL2やDockerでは解決できない場合の最終手段として考えましょう。

## 総合比較とおすすめ

私の環境（i5-1335U / 16GB RAM / 空き208GB）における最終的な判断です。

| 観点 | 第1候補 | 第2候補 |
|------|---------|---------|
| メインの第2環境 | **WSL2 + Ubuntu** | Docker Container |
| 外出先用サブ環境 | **GitHub Codespaces** | — |
| 最終手段 | — | VPS / 別PC |

### 最適解: WSL2 + Ubuntu + GitHub Codespaces の2本柱

1. **日常の第2環境 → WSL2（Ubuntu）**
   - コストゼロ、高速、完全分離
   - Ollama（ローカルLLM）にもアクセス可能
   - セットアップは15分で完了

2. **外出先からのサブ環境 → GitHub Codespaces**
   - 月120時間の無料枠で十分
   - iPadやChromebookからもアクセス可能
   - ブラウザだけで完結

この組み合わせで、ほぼ全てのユースケースをカバーできます。

## 学び・補足

- 同じマシンでOpenCodeを複数動かしたい場合、**WSL2が最も現実的な解** です。Windowsユーザーであれば今すぐ `wsl --install` することをおすすめします。
- ローカルLLM（Ollama）を使っている場合、WSL2からは `localhost:11434` でホストOSのOllamaにアクセスできます（WSL2は自動的にlocalhostをWindowsホストにルーティングします）。OpenCodeの `LOCAL_ENDPOINT=http://localhost:11434` を設定すれば連携可能です。
- GitHub Codespacesは「いつでもどこでも開発できる」という点で唯一無二の価値があります。WSL2と併用することで、環境の垣根を超えた開発フローを実現できます。
- どの方法を選ぶにしても、設定ファイルや認証情報は `.gitignore` や環境変数で適切に管理しましょう。WSL2とホストOSでGitのコミット情報が混ざらないよう、Gitのuser.emailの設定も忘れずに。

ぜひ自分に合った第2のOpenCode環境を見つけて、快適な開発ライフを送ってください。
