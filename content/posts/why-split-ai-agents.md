## 背景

先日、OpenCode上で12人のAIエージェントによる開発パイプライン「Agent Arc」を構築した話を記事にしました。

> **「12人のAIエージェントを配置する」** — opencode + deepseek 上で動作する12人のAIエージェント（Build 7人 + Blog 5人）の開発パイプラインを初めて完走させた実践レポート。
> （[前回の記事](/posts/agent-arc-first-complete-run/)より）

この記事に対して、何人かの方から同じ質問をいただきました。

> **「コードを書くエージェントもレビューするエージェントも同じDeepSeek v4 Flashですよね？同じモデルに書かせて同じモデルにレビューさせるのって、意味あるんですか？」**

確かに、直感的には疑問に思うでしょう。同じAIモデルなら、同じ視点・同じ知識で見るわけだから、自分で書いたコードを自分でレビューするのと同じじゃないか、と。

この記事では、**同じAIモデルでもエージェントを分けることに明確な意味がある**ことを、OpenCode/Claude Codeの仕組みとLLM研究の知見を交えて解説します。

## 発生した問題

「同じモデルなのにエージェントを分ける必要ってあるの？」という疑問は、一見もっともです。Agent Arcのパイプラインを具体例に挙げてみましょう。

```yaml
# .opencode/agents/backend-builder.md（一部抜粋）
mode: subagent
permission:
  edit: allow
  bash:
    "*": allow
---
<identity>
You are a senior TypeScript backend engineer.
</identity>
```

```yaml
# .opencode/agents/code-validator.md（一部抜粋）
mode: subagent
permission:
  edit: deny
  bash: deny
---
<identity>
You are a strict code reviewer. Focus on security and correctness.
</identity>
```

どちらも同じモデル（DeepSeek v4 Flash）で動作します。だったら：

- コードを書くときの思考プロセスがレビューにも引き継がれるのでは？
- 同じモデルなら同じバグの見落としをするのでは？
- エージェントを分けるのは単なるオーバーエンジニアリングでは？

この疑問は、マルチエージェントアーキテクチャに取り組む多くの開発者が一度は通る壁です。

## 原因

結論から言うと、**あなたの直感は半分正しくて半分間違っています**。

正しいのは「同じLLMモデルを裏で呼んでいる」という点。間違っているのは「だから出力も同じになる」という前提です。

### サブエージェントは独立したコンテキストで動作する

OpenCode/Claude Codeのサブエージェントは、**新しい独立したコンテキストウィンドウ**で動作します。以下のイメージを持ってください。

```
メインエージェントのコンテキスト:
┌─────────────────────────────────────┐
│ システムプロンプト（メイン）          │
│ 会話履歴（ユーザーとのやり取り）      │
│ 読み込んだファイル内容               │
│ backend-builderへのタスク依頼        │
│   → 「foo.ts にバリデーション追加」   │
└─────────────────────────────────────┘
          │
          ▼ （タスク指示＋コードのみ渡す）
          
サブエージェント（backend-builder）のコンテキスト:
┌─────────────────────────────────────┐
│ システムプロンプト（builder用）       │
│ タスク指示：「foo.ts にバリデーシ...」│
│ 読み込んだファイル内容               │
│ ツール呼び出しの履歴                │
└─────────────────────────────────────┘
```

重要なのは、**メインエージェントでの思考プロセス（中間生成物、試行錯誤、迷い）はサブエージェントに渡らない**という点です。渡されるのは「タスク指示」と「関連コード」のみ。コンテキストが完全に分離されています。

### システムプロンプトが根本的に異なる

さらに、サブエージェントにはそれぞれ**専用のシステムプロンプト**が設定されます。

**builderのシステムプロンプト（イメージ）:**
```
あなたはシニアTypeScriptエンジニアです。
与えられた要件を満たす実装を、テスト可能で保守性の高いコードで提供してください。
```

**validatorのシステムプロンプト（イメージ）:**
```
あなたは厳格なコードレビューアです。
セキュリティホール、パフォーマンス問題、型の不整合を重点的にチェックしてください。
書き込み権限はありません。発見した問題だけを報告します。
```

たったこれだけの違いで、LLMの出力は劇的に変わります。これは**Role Prompting**と呼ばれるテクニックで、同じモデルでも役割を与えるだけで出力の質が変化することが広く知られています。

### ツール権限も分離されている

builderは`read/edit/write/bash`のフル権限を持ちますが、validatorは`read/grep/glob`の読み取り専用です。

「書く側」と「チェックする側」で権限を分離することで、レビューアが誤ってコードを書き換えてしまうリスクを根本的に排除できます。これは人間の組織で「開発者とレビューアは別の人」にするのと同じ理屈です。

## 解決策

ここからが本題です。同じモデルでもエージェントを分割する効果を、4つの観点で説明します。

### 1. LLMの非決定論性（同じプロンプト ≠ 同じ出力）

LLM（大規模言語モデル）は**非決定論的**です。同じプロンプトを送っても、`temperature`パラメータ（生成のランダム性を制御する値）によって毎回異なる出力が得られます。

```
同じプロンプト × temperature 0.7
  → 出力A: 「この実装は安全です」
  → 出力B: 「この実装にはSQLインジェクションのリスクがあります」
  → 出力C: 「この実装は型安全ですが、エラーハンドリングが不足しています」
```

つまり、**コードを書いた瞬間のLLMの状態**と、**レビューする瞬間のLLMの状態**は異なります。独立したAPI呼び出しとして発行されるため、毎回異なる「視点」でコードを見ることになります。

### 2. システムプロンプトの差が出力を変える（Role Prompting効果）

先述の通り、builderとvalidatorではシステムプロンプトが根本的に異なります。役割を与えるだけでLLMの出力が変化することは、実践的にも広く知られています（Role Prompting）。たった一行の「あなたはシニアエンジニアです」でコードの品質指標が変化するケースが reports されており、システムプロンプト全体（役割・権限・ツール・出力形式）が異なるエージェントからの出力が変わるのは当然と言えます。

### 3. ツール権限の分離による規律

コードレビューエージェントが書き込み権限を持たないことで、**発見した問題を「修正する」のではなく「報告する」** という行動に自然と誘導されます。

```yaml
# ❌ 書き込み権限があると…
# レビューアが勝手にコードを直してしまう可能性

# ✅ 読み取り専用なら…
# レビューアは問題を報告することに集中
# 修正は builder の責務として明確に分離
```

この権限分離は、CI/CDのパイプラインでテスト実行環境と本番環境を分けるのと同じ設計思想です。人間のレビューと違い、**権限を徹底できる**のがAIエージェントの強みです。

### 4. コンテキストウィンドウの節約

大きなプロジェクトでは、メインのコンテキストウィンドウがファイル読み取りで埋まってしまう問題が発生します。

```python
# 例：20ファイルをレビューする場合
# エージェント分割なし → メインのコンテキストに20ファイル分が滞留
# エージェント分割あり → サブエージェント内で完結して破棄される
```

サブエージェントにファイル読み取りを任せることで、メインエージェントのコンテキストを戦略的に使えます。OpenCode/Claude Codeのコンテキストウィンドウは有限です。この節約効果はプロジェクトが大きくなるほど顕著になります。

### 研究による裏付け

「同じモデルでも独立した推論パスを複数与えると精度が上がる」という主張には、学術的な裏付けもあります。

#### Self-Consistency（自己無矛盾性）

Wang et al. (2023, ICLR) のSelf-Consistencyの研究では、**同じモデルでも複数の独立した推論パスで出力を生成し、多数決を取ることで精度が向上する**ことが示されています。例えばGSM8K（算術推論）のタスクでは、LaMDA-137Bモデルで以下の改善が確認されています。

| 手法 | GSM8K精度 |
|------|----------|
| 単一パスのGreedy Decode | ベースライン |
| Self-Consistency（k=5） | +約5% |
| Self-Consistency（k=20） | +約9% |
| Self-Consistency（k=40） | +約11% |

改善幅はモデルやタスクによって異なりますが、**独立した推論パスを複数与えること自体に効果がある**という点が重要です。コードレビューも一種の推論タスクです。builderが実装し、validatorが独立してレビューする——これはSelf-Consistencyの「独立した推論パス」をエージェント単位で実現したものと解釈できます。

#### LLM-as-Judge

「LLM自身に評価させるときは、実装とは分離した方が精度が高い」というのがLLM-as-Judgeの知見です。評価専用のプロンプト（評価基準・ルーブリック）を独立して与えることで、実装タスクとは異なる思考モードで評価できます。

#### 人間のコードレビューと同じ原理

面白いことに、これは人間のソフトウェア開発と同じ原理です。

> 自分が書いたコードのバグを見つけるのは難しい。書いたときの思考のクセが、レビュー時の視野を狭めてしまうからだ。

人間の開発現場では「コードレビューは別の人がやる」のがベストプラクティスです。AIエージェントでも同じで、**「書く人」と「チェックする人」を分ける**という設計原則は、モデルが同じでも有効に機能します。

## 学び・補足

### 3パターンの比較

エージェント設計には主に3つのパターンがあります。

| パターン | 構成 | 品質 | コスト | 速度 | 推奨シーン |
|---------|------|------|-------|------|-----------|
| **A: 分割なし** | 同じコンテキストで書かせてレビューさせる | △ | 低 | 最速 | 簡単なスクリプト、自分用ツール |
| **B: 同一モデルで分割（★推奨）** | 同じモデルでエージェントを分ける | ◯ | 中 | やや遅い | 実用的な品質が必要な場合 |
| **C: 別モデルで分割** | 別モデル（例：Claude+GPT）でエージェントを分ける | ◎ | 高 | 遅い | 最高品質が求められる本番コード |

**パターンB（同一モデル分割）が推奨**なのは、コストと品質のバランスが最も良いからです。

- パターンAは「自分で書いて自分でレビューする」状態。同じコンテキスト内なので、書いたときの思考バイアスが残る
- パターンCは確かに最高品質ですが、APIコストが2倍になり、モデルによっては待機時間も増える
- **パターンBは、コストを抑えつつ、独立したコンテキストと役割分離の恩恵を受けられる**

### 結論：エージェント分割の本質

エージェントを分けることの本質は、以下の2つに集約されます。

1. **コンテキストの独立**：
   各エージェントが独立したコンテキストウィンドウで動作することで、前のタスクの思考バイアスを受けずにタスクに集中できる

2. **システムプロンプトの特化**：
   各エージェントに最適化された役割・権限・出力形式を与えることで、Role Prompting効果を最大限引き出す

**モデルが同じでも、この2つの効果は変わりません。**

むしろ、同じモデルでエージェントを分けることのメリットは「コストを抑えながら、人間の開発チームと同じ役割分業の恩恵を受けられる」点にあります。Agent Arcでは、この設計思想で12のエージェントを定義し、実際のパイプライン実行で全テストをパスする品質を達成しています（詳細は前回の記事を参照）。

AIエージェントのパイプラインを設計するときは、「モデルを分けるべきか」ではなく「**コンテキストと役割をどのように分離すべきか**」を考えてみてください。同じモデルでも、エージェントを分割するだけで品質は確実に向上します。
